# Многопользовательская чат-система (C++ / Docker / SQLite)

Современное клиент-серверное приложение для обмена сообщениями в реальном времени, построенное на архитектуре TCP/IP с использованием Modern C++ (C++17). Проект демонстрирует навыки проектирования распределённых систем, контейнеризации, безопасного сетевого взаимодействия и организации отказоустойчивой серверной логики.

---

## Общий стек технологий

* **Язык:** C++17 (Modern C++: RAII, умные указатели, многопоточность)
* **Сборка:** CMake (Modern Presets API)
* **Контейнеризация:** Docker & Docker Compose (Multi-stage builds, Alpine Linux)
* **Сетевой стек (POSIX):**
    * `sys/socket.h` — низкоуровневое управление TCP-соединениями.
    * `netdb.h` — современное разрешение имен (DNS-aware) через `getaddrinfo`.
* **Протокол:** JSON over TCP (`nlohmann/json`)
* **Логирование:** `spdlog` (асинхронное потокобезопасное логирование)
* **База данных:** SQLite3 (персистентное хранение истории сообщений)

---

## Серверная часть (Backend)

Многопоточный TCP-сервер, выступающий в роли брокера сообщений и подсистемы хранения истории.

### Архитектурные особенности

#### Многопоточность

* Изоляция каждого клиента в отдельном `std::thread`
* Потокобезопасное управление списком клиентских дескрипторов через `std::mutex` и `std::lock_guard`

#### Постоянное хранение истории

* Автоматическое сохранение сообщений в SQLite
* При подключении нового клиента сервер загружает последние 10 сообщений (`ORDER BY id DESC`), обеспечивая сохранение контекста диалога

#### Безопасность
* **Санитизация ввода:** замена ESC-последовательностей (`\033`) на безопасные печатные символы для защиты серверного терминала
* **Защита от SQL-инъекций:** использование prepared statements (`sqlite3_prepare_v2`) и привязки параметров
* **Корректное завершение:** перехват сигнала `SIGINT (Ctrl+C)` с уведомлением клиентов, сохранением состояния и корректным освобождением ресурсов через RAII

## Клиентская часть (Frontend CLI)

Интерактивный консольный клиент с поддержкой асинхронного взаимодействия и интеллектуальной отрисовкой интерфейса.

### Ключевые возможности

#### Неблокирующий интерфейс и UX
* **Async Reception:** Разделение пользовательского ввода (`std::getline`) и приёма сетевых данных в независимые потоки. Это позволяет мгновенно отображать входящие сообщения, не прерывая процесс набора текста.
* **Smart Terminal UX:** Использование ANSI-последовательностей (`\033[2K\r`) для динамической очистки строки. При получении сообщения клиент аккуратно перерисовывает приглашение к вводу, предотвращая "разрыв" текста пользователя.

#### Сетевая гибкость
* **DNS-aware Connection:** Благодаря системному API `getaddrinfo`, клиент поддерживает подключение как по IP, так и по именам хостов. Это критически важно для работы в Docker, где связь между контейнерами идет по именам сервисов.
* **External Config:** Все параметры (хост, порт, логирование) вынесены в `config.json`, что позволяет менять среду окружения без перекомпиляции кода.

#### Надежность (Modern C++)
* **Rule of Five:** Полноценная реализация «Правила пяти» для класса клиента. Это гарантирует корректное владение сетевым дескриптором и исключает утечки ресурсов или двойное закрытие сокета при копировании/перемещении объектов.
* **RAII Management:** Автоматическое закрытие соединений и освобождение памяти при выходе объекта из области видимости.

## Развёртывание и запуск (Docker)

Проект полностью контейнеризирован. Финальные production-образы на базе Alpine Linux имеют размер менее 20 MB.

### Сборка через Docker Compose

| Действие | Команда |
| :--- | :--- |
| **Запуск сервера** | `docker compose up --build chat-server` |
| **Запуск клиента** | `docker compose run --rm chat-client` |
| **Остановка системы** | `docker compose down` |
| **Очистка образов** | `docker compose down --rmi all` |
| **Очистка сети** | `docker network rm chat_net` |
| **Просмотр логов** | `docker compose logs -f chat-server` |

---

## Ручной запуск (через Docker CLI)

| Действие | Команда | Примечание |
|---|---|---|
| **Создание сети** | `docker network create chat_net` | Создаёт «виртуальный мост» для связи контейнеров |
| **Сборка сервера** | `docker build -t my_server ./server` | Создаёт образ сервера с тегом `my_server` |
| **Запуск сервера** | `docker run -it --rm --network chat_net --name chat-server -p 9000:9000 -v $(pwd)/server/db:/app/data my_server` | Флаг `-v` сохраняет базу данных на локальном диске |
| **Сборка клиента** | `docker build -t my_client ./client` | Создаёт образ клиента с тегом `my_client` |
| **Запуск клиента** | `docker run -it --rm --network chat_net --name client_1 my_client` | Флаг `-it` позволяет вводить текст в интерактивной консоли |
| **Очистка сети** | `docker network rm chat_net` | Удаляет Docker-сеть после завершения тестирования |

## Ключевые флаги и параметры Docker

| Параметр | Назначение |
|---|---|
| `--network chat_net` | Ключевой параметр, помещающий контейнеры в одну виртуальную сеть. Без него сервер и клиент не смогут обнаружить друг друга |
| `--name chat-server` | Имя контейнера, которое используется как внутреннее «доменное имя» внутри Docker-сети. Именно это имя указывается клиентом в `config.json` |
| `-v $(pwd)/server/db:/app/data` | Проброс тома (Volume). База SQLite сохраняется на локальном диске в каталоге `server/db`, поэтому история сообщений не теряется после удаления контейнера |
| `-p 9000:9000` | Проброс TCP-порта. Позволяет подключаться к серверу извне Docker, например через `localhost:9000` |
| `--rm` | Автоматически удаляет контейнер после завершения работы, предотвращая накопление остановленных контейнеров в `docker ps -a` |

**Примечание:** В `client/config.json` адрес сервера для работы в Docker должен быть указан как `"chat_server"`.

## Локальная сборка (Modern CMake)

Если проект запускается в локальной Linux-среде или любой совместимой POSIX-системе, доступен прямой запуск без контейнеризации:

### Сервер (Server)

Перед запуском в `server/config.json` необходимо указать путь к базе данных:

```json
{
  "db_path": "chat_history.db"
}
```

Команды для сборки и запуска:

```bash
cd server
cmake --preset debug
cmake --build --preset debug
./build/chat_server
```

### Клиент (Client)

Перед запуском в `client/config.json` необходимо указать адрес локального сервера:

```json
{
  "server_ip": "127.0.0.1"
}
```

Команды для сборки и запуска:

```bash
cd client
cmake --preset debug
cmake --build --preset debug
./build/chat_client
```

## Структура проекта

Репозиторий организован по принципу **модульной монорепозиторной архитектуры**, где клиентская и серверная части развиваются независимо, сохраняя единый контракт протокола обмена.

```text
.
├── client/                       # Модуль клиента (Frontend CLI)
│   ├── include/                   # Заголовочные файлы клиента
│   ├── src/                       # Исходный код клиента (.cpp)
│   ├── CMakeLists.txt             # Инструкции сборки клиента
│   ├── CMakePresets.json          # Пресеты сборки (Debug / Release)
│   ├── Dockerfile                 # Контейнеризация клиента
│   └── config.json                # Настройки подключения к серверу
│
├── server/                       # Модуль сервера (Backend)
│   ├── include/                   # Заголовочные файлы сервера
│   ├── src/                       # Исходный код сервера (.cpp)
│   ├── db/                        # Локальная директория персистентности SQLite
│   ├── CMakeLists.txt             # Инструкции сборки сервера
│   ├── CMakePresets.json          # Пресеты сборки (Debug / Release)
│   ├── Dockerfile                 # Контейнеризация сервера
│   └── config.json                # Настройки порта и пути к БД
│
├── .clang-format                 # Единый стиль кодовой базы
├── .gitignore                    # Исключения: build/, .db, .log и временные файлы
├── docker-compose.yml            # Оркестрация всей системы
└── README.md                     # Основная документация проекта

```

## Сценарий обмена сообщениями

```text
1. Клиент запускается и считывает config.json
2. Устанавливается TCP-соединение
3. На сервер отправляется AUTH-сообщение
4. Сервер восстанавливает последние 10 сообщений из SQLite
5. Начинается двусторонняя чат-сессия в реальном времени
6. При завершении сервер рассылает событие отключения
```

## Производительность и надёжность

* Выделенный поток на каждого клиента для изоляции
* Доступ к общему состоянию защищён mutex
* Безопасное владение сокетом через Rule of Five
* Минимальный размер Docker-образа (<20 MB)
* SQLite обеспечивает долговечность сообщений
* Поддержка DNS для контейнерной оркестрации
* Корректная обработка SIGINT

## Перспективы развития

Данный проект является стабильным MVP. Ниже представлены ключевые направления для трансформации системы в высоконагруженное production-решение.

### 1. Масштабируемость и I/O модель
*   **Проблема:** Текущая модель "один поток на клиента" (thread-per-client) потребляет ~8MB стека на каждое соединение, что ограничивает лимит системы ~500-1000 одновременными пользователями.
*   **Решение:** Переход на асинхронный неблокирующий ввод-вывод с использованием системного вызова **`epoll`** (Linux) или библиотек **Boost.ASIO / libuv**. Это позволит обслуживать 10,000+ соединений силами небольшого пула потоков (Thread Pool).

### 2. Безопасность транспортного уровня
*   **Проблема:** Данные передаются в открытом виде через TCP. Любой перехват трафика (сниффинг) позволяет прочитать личные сообщения и токены.
*   **Решение:** Интеграция **OpenSSL** для реализации TLS-шифрования. Переход от сырых сокетов к защищенным SSL-сокетам.

### 3. Гарантированная доставка и Framing
*   **Проблема:** В протоколе TCP данные — это поток байтов. При высокой нагрузке возможна «склейка» JSON-пакетов или их фрагментация, что приведет к ошибке парсинга.
*   **Решение:** Внедрение системы **Message Framing**. Добавление в начало каждого пакета заголовка (Header) с указанием длины тела сообщения, чтобы `recv` точно знал, сколько байт нужно вычитать для одного корректного JSON.

### 4. Управление состоянием и авторизация
*   **Проблема:** В системе отсутствует полноценная регистрация и механизм сессий (JWT/Session Tokens). При перезагрузке клиента пользователь выбирает имя заново.
*   **Решение:** 
    *   Реализация хэширования паролей (Argon2/bcrypt) на стороне сервера.
    *   Хранение сессий в **Redis** для мгновенной проверки прав доступа без лишних запросов к основной БД SQLite.

### 5. Обнаружение обрывов (Keep-Alive)
*   **Проблема:** Если у клиента внезапно пропадет интернет (без корректного закрытия сокета), сервер может вечно держать «мертвое» соединение в списке активных.
*   **Решение:** Реализация механизма **Heartbeat (Ping/Pong)**. Сервер и клиент обмениваются короткими сигналами раз в 30 секунд; при отсутствии ответа сервер принудительно закрывает сокет и освобождает ресурсы.

### 6. Контейнерная оркестрация
*   **Проблема:** Docker Compose удобен для локального запуска, но не подходит для масштабирования на несколько серверов.
*   **Решение:** Подготовка манифестов для **Kubernetes (K8s)** и внедрение **Health Checks** (Liveness/Readiness probes) для автоматического перезапуска упавших контейнеров.
